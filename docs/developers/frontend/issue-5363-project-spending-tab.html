Issue #5363 – Project Spending Tab

Build plan & static mockup · OPRE OPS · GitHub Issue

Goal

Implement the Project Spending tab on the project detail page (/projects/:id/spending). The tab currently exists in the nav but is disabled with a Coming Soon tooltip. This plan outlines the incremental delivery approach agreed on during planning.

Priority order: (1) Wire API call + DebugCode display → (2) Agreements table → (3) Summary cards / donut chart. Summary cards are lower priority and deferred to a later sub-task.

Backend API

The endpoint already exists on the backend and is fully tested:

  • URL: GET /api/v1/projects/<id>/spending/
  • Auth: Requires JWT bearer token (Permission.RESEARCH_PROJECT GET)
  • 404: Returns 404 if project ID does not exist

Note: The response contains only aggregate data. Agreement details (name, type, start/end dates, vendor, etc.) must be fetched separately using GET /api/v1/agreements/?project_id=<id> and then filtered by FY on the frontend. The spending endpoint gives us the FY totals; the agreements endpoint gives us the metadata for the table rows.

Data Flow (Phase 1)

  1. User navigates to /projects/42/spending
  2. ProjectSpending.jsx reads :id from URL params
  3. Calls useGetProjectSpendingByIdQuery(projectId)
  4. RTK Query hits /api/v1/projects/42/spending/
  5. Response rendered inside <DebugCode data={spendingData} /> (dev-only)
  6. Placeholder headings Project Spending Summary and Agreements already visible so UX can confirm layout direction

Phase 2 addition

  1. Add local fiscalYear state (default = current FY derived from today's date)
  2. Also call useGetAgreementsByResearchProjectFilterQuery(projectId) to get full agreement objects
  3. Derive FY-filtered agreements by cross-referencing agreements_by_fy[selectedFY] IDs against the agreements list
  4. Render ProjectSpendingAgreementsTable

Key Decisions

Decision Choice Rationale
Where does FY state live? Local state in ProjectSpending.jsx Only the spending tab needs it; no need for Redux or context. Follows pattern used on CAN detail page.
Agreements table data source Two calls: spending endpoint (for FY totals) + /agreements/?project_id= (for metadata) The spending endpoint returns aggregate data only. Agreement detail (name, dates, vendor…) lives on the agreements resource. Avoids adding new backend surface for Phase 2.
Route strategy New sibling route /projects/:id/spending (not nested outlet) Consistent with existing /projects/:id approach used for tabs. ProjectDetailTabs is already navigation-based (not React Router outlets).
When to remove DebugCode When the Agreements table (Phase 2) ships DebugCode is already dev-only (guarded by import.meta.env.DEV), so it is safe to leave in during Phase 1 without impacting production users.
Summary cards priority Phase 3 (deferred) Cards depend on the spending type breakdown. The table gives more immediate user value and is more straightforward to implement. Cards can reuse DonutGraphWithLegendCard once the data shape is confirmed.

Backend Ask — FY Total Per Agreement

The FY Total column in the Agreements table currently shows — because the spending endpoint only returns project-level FY totals, not per-agreement breakdowns.

Backend team request: Please add agreements_fy_total to GET /api/v1/projects/<id>/spending/.

Proposed response shape

  "2":  { "2043": "749316710.00", "2044": "267694002.00" },
  "10": { "2043": "32289000.00" }
}

Backend implementation hint

The data is already iterated in project_list_metadata in backend/models/projects.py. A defaultdict(lambda: defaultdict(Decimal)) keyed by agreement.id → bli.fiscal_year inside the existing BLI loop would capture it with no extra DB queries.
