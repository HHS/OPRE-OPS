Issue #5275 - Project Funding Tab

Build plan & static mockup · OPRE OPS · GitHub Issue  ·  Follows phased pattern from PR #5554 (Project Spending)

Goal

Implement the Project Funding tab on the project detail page (/projects/:id/funding). The tab currently exists in the nav but is disabled with a Coming Soon tooltip. This plan covers frontend only - the backend endpoint GET /api/v1/projects/<id>/funding/ already exists and is tested.

Scope: Frontend only. Backend changes are tracked separately (see Backend Ask below). Three incremental phases, each pausing for manual testing before the next begins.

Delivery Phases

API Hook + Route + FY Selector + DebugCode

Backend API

The endpoint already exists and is fully tested:

  • URL: GET /api/v1/projects/<id>/funding/?fiscal_year=YYYY
  • Auth: Requires JWT bearer token (Permission.RESEARCH_PROJECT GET)
  • fiscal_year param: Optional - defaults to current fiscal year on the backend if omitted
  • 404: Returns 404 if project ID does not exist

Key Decisions

DecisionChoiceRationale
Where does FY state live? Local state in ProjectFunding.jsx Follows CAN funding convention - only the funding tab needs it; no Redux or context required.
FY selector default getCurrentFiscalYear() - no fallback to highest available FY Matches Can.hooks.js pattern exactly. Show zeros/empty if current FY has no data.
FY change triggers re-fetch? Yes - RTK Query re-fetches with new fiscal_year arg API is FY-scoped for portfolio/CAN/cans data. RTK Query caches per arg so switching back is instant.
Portfolio abbreviations Interim: reverse-lookup from PORTFOLIO_ORDER constants by portfolio name API does not return abbreviation. Reverse-lookup by full name is pragmatic short-term. Backend ask filed to add abbreviation to response long-term (see below).
"FY N New Funding" label FY {selectedFY} New Funding Backend returns new_funding as money appropriated in the selected FY. "FY 2024" shown in Figma was a mockup artifact.
"Project Funding By FY" bar chart range Last 5 FYs relative to selected FY, derived client-side Matches CANBudgetByFYCard pattern (getLast5FiscalYears). Uses constants.budgetsByFYChartColors. Zeros shown for years with no data.
Empty state (no funding data) Show cards/table with zero values; "No CANs found for this project" in table body Follows CAN funding convention - no special empty-state screen, just graceful zeros.
Active Period display format active_period === 1 ? "1 Year" : `${active_period} Years"` Matches Figma ("5 Years", "1 Year"). API returns a plain integer.
Route strategy New sibling route /projects/:id/funding Consistent with /projects/:id/spending and existing ProjectDetailTabs navigation pattern.
When to remove DebugCode End of Phase 3 Already dev-only - safe to leave during Phases 1-2 without impacting production users.

Data Flow

Phase 1

  1. User navigates to /projects/42/funding
  2. ProjectFunding.jsx reads :id from URL params
  3. Calls useGetProjectFundingByIdQuery({ id: projectId, fiscalYear })
  4. RTK Query hits /api/v1/projects/42/funding/?fiscal_year=2025
  5. Response rendered inside <DebugCode data={fundingData} /> (dev-only)
  6. FY selector visible; changing it updates selectedFiscalYear state and triggers a new (or cached) RTK Query fetch

Phase 2 additions

  1. On data load, derive fyChartData: call getLast5FiscalYears(selectedFY), map each year to funding_by_fiscal_year (zero if absent)
  2. Derive portfolio bar segments: map funding_by_portfolio items through getPortfolioAbbrevAndColor(name) helper that reverse-looks up PORTFOLIO_ORDER; pass through computeDisplayPercents
  3. Derive carry-forward/new segments from funding_by_can; pass through computeDisplayPercents
  4. Render ProjectFundingByPortfolioCard (full width), then two-column grid of ProjectFundingByCANCard + ProjectFundingByFYCard

Phase 3 additions

  1. While isFundingLoading is true, render ProjectFundingCANsTableLoading skeleton (column layout preserved)
  2. On data arrival, render ProjectFundingCANsTable from fundingData.cans
  3. Format active_period: 1 → "1 Year", N → "N Years"; null/undefined → TBD
  4. Dynamic column header: "FY {twoDigitYear} Project Funding" (e.g. "FY 25 Project Funding")
  5. Remove <DebugCode>

Backend Ask — Add abbreviation to Portfolio Funding Response

The funding_by_portfolio array currently returns portfolio (full name) but not the portfolio abbreviation. The frontend derives the abbreviation by reverse-looking up PORTFOLIO_ORDER constants by name - a fragile coupling that will break if portfolio names drift between backend data and frontend constants.

📋 Backend Story (using feature_work_template.md)

User Story
As a frontend developer, I want the GET /api/v1/projects/<id>/funding/ response to include the portfolio abbreviation in each funding_by_portfolio item so that I can display and color-code portfolio segments in the horizontal stacked bar without relying on client-side name matching.

Acceptance Criteria

  • funding_by_portfolio items include an abbreviation field (e.g. "CC", "CW", "HS")
  • ProjectFundingByPortfolioSchema updated to serialize abbreviation
  • Existing backend tests updated; no change to other fields

Tasks

  • Add portfolio.abbreviation to get_project_funding() in backend/models/projects.py
  • Add abbreviation = fields.String() to ProjectFundingByPortfolioSchema in backend/ops_api/ops/schemas/projects.py
  • Update frontend ProjectFundingByPortfolioCard to use response abbreviation instead of reverse-lookup (follow-up frontend PR)

Additional Context

  • The Portfolio model already has an abbreviation field - this is a minor schema addition
  • Frontend workaround is in place (reverse-lookup via PORTFOLIO_ORDER) until this lands
